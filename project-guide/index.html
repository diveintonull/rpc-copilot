01

先看懂：这个项目到底是什么

先建立正确心智模型，再读代码。

一句话定义 GRC Copilot 是一个“证据优先的合规工作流 Agent”：它从受治理的法规语料中找证据， 再完成法规问答、条款比较或控制差距分析，并尽量让每个结论都能回到具体版本和条款。
Input

用户问题

法规要求是什么、两部法规有什么差异、企业当前控制存在哪些差距。

Core

受控工作流

系统按任务模式走固定 Graph，不让模型自由决定所有步骤。

Output

回答 + 证据 + Trace

用户不仅看答案，还能看到原文证据、版本和可观察执行节点。

它为什么不是普通聊天机器人

  • 回答前有检索：模型不能只靠参数记忆回答法规问题。
  • 证据带版本：GBT-22239@2019#7.1.4.1 同时包含来源、版本和条款。
  • 工作流有限制：缺少证据、缺少比较一侧或缺少企业现状时应拒答。
  • 回答后有校验：引用编号、版本一致性和语义支持会被再次检查。
  • 失败也是结果：项目把拒答和失败案例当成评测对象，而不是只展示成功 Demo。

它又为什么不算“自主智能体”

真实 API 根据前端传入的 mode 固定选择工作流；工具调用也主要由 Graph 节点决定。 所以它更像受治理的 LLM 工作流，而不是让模型自由规划、动态选择任意工具的通用 Agent。 在合规场景中，这种保守设计通常比“更自主”更合理。

02

一条请求从头到尾怎样运行

用“三级系统身份鉴别要求”串起所有模块。

示例请求 用户在“法规问答”模式输入：GB/T 22239—2019 对三级系统的身份鉴别提出了哪些要求？

  1. 前端发送请求

    web/app.js 生成 request ID，POST 到 /chat，请求体包含 mode、query 和 control_text。

  2. API 注册任务

    api/main.py 用 TaskManager 防止同一 request ID 重复执行，并建立 SSE 事件队列。

  3. RealAgentRunner 选择 Graph

    api/real_runtime.py 根据前端 mode 选择已经编译好的法规问答 Graph，初始化 AgentState。

  4. 匹配并加载 Skill

    意图映射到 regulation-qa，只读取这一份 SKILL.md 正文，作为领域 SOP 注入模型。

  5. 工具检索法规证据

    search_regulation 把问题编码成向量，在 Qdrant 搜子块，扩展成完整父条款，再用 Reranker 排到前五。

  6. LLM 根据编号证据回答

    证据被包装成 [1]、[2]……模型被要求每个事实句都附上最小必要引用。

  7. 引用验证与一次修复

    系统检查未知引用、版本不一致、无引用声明和不受证据支持的声明；首次失败时可尝试修复一次。

  8. SSE 流式返回

    回答 token、证据卡片、Trace 节点和终态分别作为 SSE 事件发送，前端按到达顺序更新三个区域。

重要：两条“流”不要混淆 数据流描述 query、evidence、answer 怎样变化；控制流描述 Graph 下一步走哪个节点。 Agent 工程通常就是同时管理这两条流。

目录与代码地图

每个目录只回答一个核心问题。

ingest/法规原文怎样变成有版本、有父子关系的可检索数据。
rag/问题怎样检索、融合、精排，以及答案引用怎样校验。
agent/状态、工具、Skills、LLM 适配器和 LangGraph 节点怎样编排。
api/怎样把同步 Graph 包装成 FastAPI、SSE 和任务生命周期。
mcp_server/怎样把确定性工具暴露成标准 MCP 工具。
evals/数据集、指标、消融和最终评测怎样运行。
tests/用 fake、依赖注入和契约测试验证每个边界。
web/不依赖前端框架的 SSE 客户端、Markdown 渲染和可观察 UI。
skills/三种任务的工作步骤、证据规则、输出格式和拒答边界。
data/parsed/纳入版本控制的五份解析语料和父章节存储。

推荐的代码阅读顺序

  1. agent/state.py：先认识整个系统传递的数据。
  2. agent/graph.py：看控制流和所有节点连接。
  3. agent/nodes.py：看每个工作流实际做什么。
  4. agent/tools.py：看 Agent 能调用哪些确定性能力。
  5. rag/retrieve.py：看检索流水线。
  6. ingest/chunk_parent.py 与 ingest/index.py：倒推证据怎样生产。
  7. api/real_runtime.py 和 api/main.py：看如何接入服务。
  8. evals/run_eval.py：最后看系统怎样被测量。
04

语料解析与父子切分

切分不是把字符串随便每 500 字砍一刀。

第一步：把不同格式统一成 Markdown

PDF

GDPR

parse.py 使用 pymupdf4llm 处理文本型 PDF。

OCR

GB/T 标准

parse_mineru.py 使用 MinerU，解决中文 PDF 字体映射乱码。

HTML

中国法律

parse_html.py 提取正文容器，再转为 Markdown。

第二步：建立版本化身份

ingest/schema.py 定义 DocumentMeta。一个文档不是简单叫“网络安全法”， 而是用 source_id + version 形成 document_id。 再加条款号形成父章节 ID： 

cybersecurity-law@2025-amended#第二十三条
└──── source_id ────┘ └─ version ─┘ └ section ┘

GBT-22239@2019#7.1.4.1

这样做的意义是：同一法规的不同版本不能在向量库里悄悄覆盖或混在一起。

第三步：先切父章节，再切子块

父章节 Parent

一个完整、可引用的法规条款。例如 7.1.4.1 的全部正文。

  • 不直接做向量搜索
  • 保存在 _parents_store.json
  • 最终作为完整上下文返回

子块 Child

父章节内部的短窗口，默认 500 字、重叠 100 字。

  • 生成 embedding
  • 写入 Qdrant
  • 携带 parent_id 指针
完整条款900 字
Child 00–500
Child 1400–900
搜索命中Child 1
向上扩展完整 Parent

为什么搜索小块，却返回大块

  • 小块更容易匹配：主题集中，不会被一整章无关文字稀释 embedding。
  • 大块更适合回答：返回完整条款，避免缺少前提、例外和上下文。
  • 引用更稳定：Gold 和答案都引用父章节，而不是不稳定的向量点编号。
自测：为什么不直接把父章节也放进 Qdrant？
可以，但长章节的向量会混合多个语义，召回通常不如短子块；同时如果最终只返回子块，又容易截断要求。 当前方案用 child 解决“找得准”，用 parent 解决“读得全”。
05

检索、融合与 Rerank

召回负责“别漏掉”，精排负责“把正确的放前面”。

项目实现过的四个部件

Dense

BGE-M3 向量检索

适合语义近似，例如口语问题与法规正式表述不完全相同。

Sparse

Jieba + BM25

适合精确词、条款号和关键词，但目录文本也可能得到高分。

Fusion

RRF

根据名次而不是原始分数合并 Dense 与 Sparse 排名。

Rerank

CrossEncoder

同时阅读 query 和候选正文，成本更高，但排序判断更细。

真实默认路径

Agent 的默认工具不是“所有高级功能全开”，而是根据消融实验选择： 父子分块 + Dense + Parent Expansion + Rerank，Sparse 默认关闭。

Query用户问题
EmbeddingBGE-M3
Dense Top 20子块候选
Parent Expand去重父条款
Rerank Top 5完整条款
Evidence交给 Agent

为什么 Sparse 做了却默认关闭

因为工程选择应该由整体数据决定，而不是由“听起来先进”决定。 在父子切分下，加入 Sparse 后 Recall@5 从 81.91% 降到 80.85%，MRR 也下降。 这说明当前语料和 tokenizer 组合下，BM25 带来的候选并没有改善整体结果。

核心代码定位
  • rag/dense.py::search_dense：查询向量与 Qdrant filter。
  • rag/sparse.py::SparseIndex：中文分词和内存 BM25。
  • rag/fusion.py::reciprocal_rank_fusion：稳定 RRF 排名。
  • rag/rerank.py::rerank_hits：CrossEncoder 与失败回退。
  • rag/retrieve.py::retrieve：统一流水线入口。
  • agent/tools.py::_default_search：Agent 实际选择的配置。
06

LangGraph Agent 怎样工作

Graph 的价值是显式管理状态、分支、重试和终态。

AgentState：所有节点共享的数据合同

request_id        请求身份
query             当前检索问题，重试时可能被重写
control_text      企业当前控制事实
intent            regulation_qa / clause_comparison / gap_analysis / unsupported
active_skill      当前选择的 Skill
skill_text        已加载的 SOP 正文
tool_calls        可审计的工具调用记录
evidence          法规证据或 gap rows
answer            当前答案
citations_valid   校验结果
retry_count       已重试次数
final_status      completed / refused / cancelled / failed
trace             节点事件列表

Graph 控制流

route_intent取得模式
match_skill匹配元数据
load_skill加载正文
check_cancel取消边界
execute三选一工作流
verify验证结果
retry / finish一次重试或终止

三种工作流分别做什么

Regulation QA

法规问答

搜索证据 → 编号 → 生成回答 → 验证每个事实句。首次引用失败可用同一批证据修复答案。

Comparison

条款比较

模型先规划左右来源；精确定位足够时调用 get_clause，否则分别检索两侧。缺任意一侧就拒答。

Gap Analysis

控制差距

从 control_text 提取事实 → 检索要求 → 映射 aligned/partial/gap/unknown → 要求人工确认。

为什么只允许重试一次

没有上限的 Agent 循环可能重复烧 Token、反复调用工具，甚至永远无法终止。 当前 Graph 的策略是：第一次验证失败时，法规问答优先修复引用；其他失败可重写检索 query。 第二次仍失败就拒答。Graph 还设置 recursion_limit=16，形成双重上限。

工程理解 LangGraph 在这里不是为了“显得 Agent 化”，而是把本来散落在 if/else 中的生命周期变成可测试的状态机。
07

渐进式 Skills 是什么

先只看目录，选中后才加载完整 SOP。

一个 Skill 包含两层信息

Frontmatter：轻量目录

name: regulation-qa
description: 何时使用、何时不使用

启动时读取，供系统选择；不加载大量正文。

Body：完整工作方法

包含 Required inputs、Workflow、Evidence rules、Output、Boundary and refusal。

只在匹配当前任务后读取。

渐进式加载过程

Discover扫描 */SKILL.md
Catalog只存名称与描述
Matchintent → skill
Load读取一份正文
Inject加入 system prompt

为什么不一次加载全部 Skill

  • 浪费上下文：三个 SOP 都会占 prompt token。
  • 产生冲突：比较、问答和 gap analysis 的输出规则不同。
  • 误触发：不支持的请求也可能收到不相关领域指令。
  • 可观察性差：很难判断哪份说明真正影响了答案。
不要神化 Skills 当前 Skills 本质上是结构化 Prompt SOP，加上安全的发现与按需加载器。 它不等于模型训练，也不是模型自动学习了新能力。
08

Tools 与 MCP

把可确定执行的能力从 LLM 中拆出来。

三个领域工具

Search

search_regulation

按 query 搜索证据，可限制 source_ids。

Lookup

get_clause

用 source_id、version、section_number 精确查一个父条款。

Compare

compare_clauses

分别精确查左右两侧，保持两侧 Evidence 不混合。

为什么 get_clause 不需要 LLM

如果用户已经给出完整定位符，最安全的行为是精确查字典，而不是让模型“理解一下大概是哪条”。 工程上应遵守：能用确定性逻辑完成的步骤，不交给概率模型。

Local 与 MCP 后端

LocalToolBackend

同一 Python 进程直接调用函数，默认部署使用，延迟低，调试简单。

MCPToolBackend

通过标准 MCP 协议调用相同工具，便于被外部 Agent 客户端使用，并返回结构化错误。

mcp_server/server.py 用 Pydantic 定义输入输出，标记工具只读、幂等、非破坏性。 agent/tools.py 再把 MCP 响应适配回 AgentToolBackend 接口。

当前 MCP 的工程边界 默认 stdio caller 每次工具调用都会建立短生命周期会话，不适合高并发生产环境；真实 API 默认也仍使用 LocalToolBackend。
09

生成、引用校验与安全边界

“带引用”不等于“引用正确”。

证据怎样进入 Prompt

每条 Evidence 先被编号，并且正文中的 HTML 边界字符会被转义：

[1] parent_id=GBT-22239@2019#7.1.4.1
source_id=GBT-22239
version=2019
section_number=7.1.4.1
<evidence>
完整法规条文……
</evidence>

如果法规正文中恶意包含 </evidence> 和“忽略系统指令”，转义后也不能提前关闭外层证据容器。 这是一种文档 Prompt Injection 缓解措施，但不是完整安全证明。

引用验证分两层

确定性检查

  • 引用编号是否存在
  • parent_id 中的版本是否等于 evidence.version
  • 事实声明是否完全没有引用

语义检查

  • 被引用证据是否直接支持 claim
  • claim 是否比证据范围更宽
  • 多证据联合是否真正支持比较结论

一个校验例子

Pass

“应定期更换鉴别信息[1]”

证据明确包含“定期更换”，引用编号也有效。

Fail

“管理员必须每天换密码[1]”

证据只说定期，没有“每天”，属于 materially broader。

Fail

“管理员必须每天换密码”

没有任何编号引用，先在确定性层失败。

为什么语义校验仍不可靠

当前语义判定仍由 LLM 完成，通常与生成模型来自同一端点。它比完全不检查好，但不是独立的事实证明。 更强方案包括独立 NLI 模型、规则校验、人工抽检、多评委一致性以及针对业务字段的确定性验证。

10

API、SSE、任务与前端

把同步、耗时的 Agent 工作包装成用户可观察的服务。

为什么用 SSE

浏览器只需要服务器单向推送事件，SSE 比 WebSocket 更简单；事件是文本协议，天然适合 token 流、引用和 Trace。 后端定义七种事件：status、text、reference、recommendation、trace、done、error。

POST /chat创建请求
TaskManager注册任务
Worker Thread运行 Graph
Queue跨线程投递事件
SSE逐帧编码
Browser增量渲染

中间状态为什么能提前展示证据

RealAgentRunner.run_streaming 使用 LangGraph 的 stream_mode="values"。 每得到一个新状态，就让 emitter 比较 evidence 和 trace 是否增加；新证据不必等最终答案结束才能显示。

前端没有框架，仍然做了什么

  • 自己实现可处理碎片化 CRLF frame 的 SSE parser。
  • 按 request_id 隔离旧请求和新请求事件。
  • 忽略终态之后到达的迟到事件。
  • 用 textContent 和 DOM 节点渲染 Markdown，避免直接 innerHTML 注入。
  • 让答案中的 [1] 成为按钮，可以定位右侧证据卡片。
  • Trace 只展示白名单字段，不泄漏完整 Prompt、Skill 或隐藏推理。
11

测试到底怎么测

测试重点不是调用真实大模型，而是验证工程合同。

当前结果 本机包含未提交测试时是 302 passed；只运行 Git 已跟踪测试是 301 passed。 这也说明测试数字必须在 clean clone 中复现，不能只看本机输出。

为什么大量使用 Fake 和依赖注入

真实 embedding、Qdrant 和远程 LLM 又慢、又贵、还可能波动。 单元测试把它们替换成可预测的 fake，专门检查“如果依赖返回这些值，业务逻辑是否正确”。

测试层典型文件主要证明不能证明
纯函数test_chunk.py / test_metrics.py窗口、重叠、指标数学正确真实检索质量
Schematest_schema.py / test_eval_schema.py非法字段、版本 ID 被拒绝法规内容正确
Graphtest_agent_graph.py分支、重试、取消边界、终态真实 LLM 能否稳定遵守 Prompt
工具协议test_agent_tools.py / test_mcp_server.pyLocal/MCP 返回相同领域结构MCP 高并发性能
API 契约test_api.pySSE 顺序、终态、断连、任务清理工作线程真正被中断
部署组合test_deployment.pydemo/real 模式选择与 readiness真实 Docker 环境可启动

一个典型 Graph 测试怎样写

Arrange：注入 FakeTools、FakeLLM、FakeCancellationChecker
Act：     graph.invoke(initial_state)
Assert：  tool_calls 数量正确
         trace 节点顺序正确
         retry_count 不超过 1
         final_status 等于 refused / completed / cancelled

怎样理解“302 个测试都过了”

它说明实现满足作者写下的合同，没有明显回归；它不说明答案语义正确率达到生产要求。 测试回答的是“代码有没有按设计工作”，评测回答的是“这个设计实际效果怎么样”。

自测：为什么不能把评测集 60 条都写成单元测试并宣布效果稳定？
因为远程模型和真实检索包含概率性与环境依赖。把输出硬编码成单测只会验证旧输出没有变化， 不能判断新答案是否同样正确。质量评测需要指标、冻结配置、失败保留和人工复核。
12

评测数据与指标怎么读

先分清检索指标、生成指标和系统指标。

60 条数据集包含什么

29法规问答
12条款比较
6控制差距
13应拒答请求

每条 case 有 question、task_type、gold_points、gold_citations、should_refuse、source_versions 和 tags。 Gold 字段只允许评分器看到，不能提前泄漏给正式路由与回答模型。

Gold citation 指标的局限

如果答案引用了一个真实相关、但标注者没有放进 gold 的条款，它也会被计为 false positive。 因此 citation precision 不是纯粹的“事实正确率”。但 36.70% 仍是明显警报，因为失败分析确实发现了引用相似但错误安全等级条款的案例。

一次失败的调优也有价值

最终评测曾尝试把 generation_context_k 从 5 减到 3，希望减少错误引用。 Citation precision 只增加约 0.18 个百分点，coverage 却下降约 10.64 个百分点，因此拒绝该配置。 调优的目标不是“必须找到一个更好数字”，而是保留能解释的负结果。

调优方法模板 先定义最弱指标 → 提出一个只改变一个变量的假设 → 固定数据、模型和其他参数 → 运行完整样例 → 同时看收益与代价 → 接受或拒绝 → 保留失败案例。

真实缺陷与工程边界

真正理解项目，必须能解释它哪里还不成立。

Runtime

任务停止不等于终止推理

取消 asyncio 外层任务不会终止 to_thread 中已经运行的模型调用。

Agent

生产运行时不是自主路由

前端 mode 固定 Graph；最终评测中的 LLM router 与生产 API 不是同一条链。

Validation

Gap 校验不够深

主要检查 evidence 非空、人工确认和少量黑名单，没有逐行验证 requirement 与 current_state。

Index

索引没有版本迁移

启动只检查 collection 非空，不比较语料 hash、模型版本和索引 schema。

Evaluation

评测与生产 Graph 分离

旧报告绕过当前 Graph 校验和修复逻辑，不能证明最新端到端质量。

Repository

公开证据不完整

本地 results/docs 被忽略，README 的 302 tests 依赖一个未提交测试。

如果要接近生产级，改进顺序

  1. 先让 GitHub clean clone 可以复现测试、评测与 README 数字。
  2. 让最终评测直接调用当前生产 Graph，并冻结干净 commit。
  3. 逐行验证 gap requirement、current_state、classification 和 evidence。
  4. 传播真正的 cancellation token，并能停止或关闭模型流。
  5. 为索引增加 corpus hash、embedding model version 和原子切换。
  6. 增加 CI、Docker 集成 smoke、限流、认证、并发和持久化观测。

每读一个模块，都问四个问题

Contract

输入输出是什么？

看 TypedDict、dataclass、Pydantic model 和函数签名。

Boundary

什么情况下拒绝？

空证据、版本冲突、缺控制文本、依赖失败分别怎样处理。

Test

哪个测试证明它？

不要只看注释；找到能让实现失败的断言。

Tradeoff

代价是什么？

质量、延迟、Token、复杂度和维护成本之间怎样取舍。

面试复述框架

按“问题—设计—证据—边界”讲，而不是报技术名词。

90 秒项目介绍

我做了一个证据优先的 GRC 合规工作流 Agent，重点不是让模型回答得像专家， 而是让法规结论能追溯到具体来源、版本和条款。语料侧使用父子分块： 短子块做 BGE-M3 检索，命中后返回完整父条款，再由 Reranker 精排。 Agent 侧用 LangGraph 编排法规问答、条款比较和控制差距分析，并在生成后检查引用编号、 版本和语义支持，失败最多修复一次。服务侧用 FastAPI 和 SSE 分别流式展示回答、证据和安全 Trace。 我还构建了 60 条评测集和八组检索消融，父子 Dense + Rerank 的 Recall@5 为 81.91%。 当前主要不足是引用精度、真实取消和评测与生产 Graph 尚未完全统一。

高频追问与回答方向

为什么用父子分块，不直接固定窗口？
短块提高语义召回，完整父条款保留前提与例外；同配置消融中 Recall@5 从 50.00% 提升到 69.15%。
为什么实现了 BM25 却没默认开启？
在当前 60 条集合和中文语料上，父子 Hybrid + Rerank 的整体 Recall@5/MRR 低于父子 Dense + Rerank，所以依据消融关闭，而不是依据组件数量。
你的 Agent 到底自主在哪里？
生产运行时是用户选择受控工作流，模型只在比较规划、控制提取、生成和语义判断中参与；这是为了降低合规场景的不可预测性，不应包装成通用自主 Agent。
302 个测试为什么不能证明质量？
测试主要验证状态机、协议和边界合同，真实答案质量由 60 条评测和人工失败分析衡量；两者解决不同问题。
项目最危险的失败是什么？
不是明确拒答，而是引用了内容相似但安全等级错误的条款，答案看起来专业且有引用，用户更容易误信。
如果再做一周，优先修什么？
先统一生产 Graph 与评测器并在干净 commit 重跑，再做 gap 逐行语义验证和真正的模型取消，最后补索引版本化与 Docker 集成 CI。
面试原则 你可以承认项目大量借助 AI 完成，但必须能解释每个关键选择、复现数字、指出缺陷，并现场沿着一条请求定位代码。 不理解却背结论，比直接承认还没掌握更危险。

项目术语速查

忘记名词时回来查，不需要死记。

Parent Section
完整可引用法规章节，保存在 JSON store。
Child Chunk
父章节内部用于 embedding 和向量搜索的短块。
Dense Retrieval
基于 embedding 向量相似度的语义检索。
Sparse Retrieval
基于词频和逆文档频率的 BM25 检索。
RRF
Reciprocal Rank Fusion，用排名倒数融合多路候选。
Reranker
同时阅读 query 和 document，重新计算相关性。
Evidence
带来源、版本、条款号、正文和分数的稳定结构。
AgentState
LangGraph 所有节点共享并逐步更新的数据合同。
Progressive Skill
先暴露轻量目录，匹配后只加载一份完整 SOP。
Entailment
判断证据是否直接支持某个回答声明。
Recall@K
前 K 个检索结果覆盖了多少 Gold 证据。
MRR
第一个相关结果排名倒数的平均值。
Citation Precision
答案引用中有多少属于标注 Gold。
Citation Coverage
Gold 引用中有多少被答案实际使用。
SSE
服务器向浏览器单向推送文本事件的 HTTP 协议。
MCP
让 Agent 客户端以统一协议发现和调用工具。
